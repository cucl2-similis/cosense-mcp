- **ドキュメントの同期確認**  
  実装中にAIが判断・補完した未定義の仕様や例外ケースに気づいた場合、  
  ドキュメントへの反映が必要かをユーザーに確認してから更新する。  
  ドキュメントの無断生成・変更は行わない。

- **テスト駆動開発（TDD）の徹底**  
  本プロジェクトはTDD（Test-Driven Development）で開発する。  
  実装に着手する前に、対象機能のテスト仕様書（`documents/test-spec.md`）を作成または更新し、テストケースの設計をユーザーと合意してからテストコードの作成に入る。  
  テストコードは仕様書のテストケースに基づいて作成し、テストが失敗することを確認してから実装に着手する（Red → Green → Refactor）。  
  仕様書やテスト設計に修正・変更を加えた場合は、必ず対になるテストコード側にも即座に反映させるものとする。

- **ドキュメントの視認性維持（可読性ルール）**  
  箇条書きは先頭に太字タイトルと本文を1行で凝縮するスタイルを避け、タイトルの直後に「半角スペース2つ」を付与して改行する。  
  全体の文字数は1項目あたり120文字程度を目安とし、1行が長すぎる場合のみ意味の切れ目で適度な改行を入れるものとする。

- **コミットメッセージの規格化**  
  コミットメッセージは必ず「Conventional Commits」の仕様に準拠し、  
  Prefix（`feat:`, `fix:` 等）に続く説明文はすべて日本語で記述するものとする。  
  対応するIssue番号はタイトル行には含めず、フッターに `Refs: #12` の形式で記載する。  
  コミットメッセージでは `Closes` 等の自動クローズキーワードは使用しない。自動クローズはPRひな型の `Closes` により行う。  
  AIと共同で作業したコミットには必ず `Co-authored-by` をフッターに記載する。

  ```
  feat: 検索ツールを実装

  （任意）変更の意図や補足を本文に記載する。

  Refs: #12
  Co-authored-by: Claude <noreply@anthropic.com>
  ```

- **テスト実行コマンドと事前環境設定**  
  テストは必ず `./mvnw clean test` で実行する。  
  PowerShellから実行する場合は、実行前に `$env:JAVA_HOME` が適切なバージョンのJavaのJDKパスに設定されていることを確認する。  
  未設定の場合は `$env:JAVA_HOME = "<JDKのインストールパス>"` を設定してから実行する。  
  この設定を省略すると "JAVA_HOME is not defined correctly" エラーで即終了するため、エラー調査に入る前に設定済みかを最初に確認する。

- **MCPサーバとしての動作確認**  
  本プロジェクトはMCPサーバであるため、JUnitテストの通過に加えて、MCP Inspector による実機確認を行う。  
  以下の手順をユーザーに案内し、確認完了の報告を受けてから対応完了とする。
  1. `./mvnw clean package` でJARをビルドする。
  2. 以下のコマンドでMCP Inspectorを起動する。  
     `npx @modelcontextprotocol/inspector %JAVA_HOME%/bin/java -jar ./target/mcp-cosense-1.0.0.jar`
  3. ブラウザで開いたMCP Inspector上で、`cosense_search_pages` と `cosense_get_page` の両ツールが一覧に表示されること、および実際に呼び出してレスポンスが返ることを確認する。

- **ブランチ・Issue・PRによる開発フロー**  
  GitHub上の操作（Issue作成・PR作成等）は、`gh`（GitHub CLI）を使って行う。  
  開発計画を立てたら、機能・改善・バグ修正など独立して完結する作業の単位でGitHub Issueを先行作成する（例：「検索ツールの実装」「エラーハンドリングの追加」）。  
  各Issueの対応は `develop` ブランチから `{種別}/issue-{Issue番号}-{簡潔な英語説明}` の形式でブランチを切って開始する。  
  種別は `feature`（機能追加）・`fix`（バグ修正）・`docs`（ドキュメント）・`refactor`（リファクタリング）などを使用する（例：`feature/issue-12-add-search-tool`）。  
  ブランチ名は英数字・ハイフン・スラッシュのみを使用し、日本語や記号は含めない。  
  コミットはテストコードと実装コードを分けて行う（例：`test: 検索ツールのテストを追加` → `feat: 検索ツールを実装`）。  
  各コミットの前には必ず `./mvnw clean test` を実行し、全件成功することを確認してからコミットする。  
  すべてのコミットが完了したら `develop` へのマージPRを作成して対応を完了とする。

  **Issueひな型**（`gh issue create` で以下のbodyを使用する）

  ```markdown
  ## 概要

  <!-- この Issue で対応する内容を1〜2文で説明する -->

  ## 対応内容

  <!-- 実施する作業を箇条書きで列挙する -->

  -

  ## 完了条件

  <!-- この Issue が完了したと判断できる条件を列挙する -->

  - [ ] テストコードを作成し、Red 状態を確認した
  - [ ] 実装コードを作成し、テストが Green になることを確認した
  - [ ] `./mvnw clean test` が全件グリーンで通過した
  ```

  **PRひな型**（`gh pr create --base develop` で以下のbodyを使用する）

  ```markdown
  ## 関連Issue

  Closes #

  ## 概要

  <!-- このPRで対応した内容を1〜2文で説明する -->

  ## 対応内容

  <!-- 実施した作業を箇条書きで列挙する -->

  -

  ## テスト結果

  - [ ] `./mvnw clean test` が全件グリーンで通過した
  - [ ] MCP Inspector で全ツールの動作を確認した

  ## レビュー観点

  <!-- レビュアーに特に確認してほしい箇所があれば記載する -->
  ```
